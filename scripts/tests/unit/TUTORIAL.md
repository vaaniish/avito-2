# Руководство по unit-тестам

## Что здесь есть

- `order-status-fsm.unit.test.ts` — инварианты переходов статусов заказа
- `anti-circumvention.unit.test.ts` — сигналы увода сделки
- `session-token.unit.test.ts` — подпись, проверка, истечение и production-ограничения

## Когда запускать

- После любых изменений в доменной логике
- Перед integration и e2e, чтобы быстро поймать базовые регрессии

## Как запускать

```bash
npm run test:unit
```

## Как анализировать

- Unit-тесты проверяют функции, а не UI
- Основной сигнал — полный `PASS` без flaky-поведения
- При `FAIL` смотреть:
  - имя теста
  - `AssertionError`
  - expected/actual
  - файл и строку в стектрейсе

## Практика

- Unit-тесты должны быть маленькими, быстрыми и стабильными
- На каждую новую бизнес-ветку полезно добавлять 1-2 точечных unit-теста
- На каждый найденный баг: сначала тест, потом фикс
