1. Уведомления для продавцов


   * Проблема: Продавцы не получают уведомлений, когда покупатель задает вопрос по их товару. Им приходится вручную проверять каждую
     свою позицию, что неудобно и замедляет коммуникацию.
   * Рекомендация: Создать новую модель Notification в schema.prisma:


    1     model Notification {
    2       id          Int      @id @default(autoincrement())
    3       user_id     Int      // ID получателя (продавца)
    4       user        AppUser  @relation(fields: [user_id], references: [id])
    5       type        String   // "NEW_QUESTION", "NEW_ORDER" и т.д.
    6       message     String
    7       target_url  String   // Ссылка на страницу товара или заказа
    8       is_read     Boolean  @default(false)
    9       created_at  DateTime @default(now())
   10     }
      При создании нового вопроса (ListingQuestion) также создавать запись в этой таблице для продавца. Затем на фронтенде в Header.tsx
  добавить иконку "колокольчик", которая будет запрашивать API на наличие непрочитанных уведомлений и отображать их в выпадающем
  списке.


  2. Ответы на вопросы


   * Проблема: Продавцы не могут отвечать на вопросы покупателей. Сейчас поле answer в модели ListingQuestion можно заполнить только
     напрямую в базе данных, что делает его бесполезным.
   * Рекомендация: В личном кабинете партнера создать новую страницу PartnerQuestionsPage.tsx. Она будет выводить список всех вопросов
     к товарам продавца. У каждого вопроса должно быть поле для ввода ответа и кнопка "Ответить". Для этого нужно создать новый
     защищенный API-маршрут (например, PATCH /api/partner/questions/:questionId/answer), который позволит продавцу обновлять поле
     answer в ListingQuestion.

  3. Логика расчета рейтинга


   * Проблема: Сейчас рейтинг объявления (rating в MarketplaceListing) — это просто статичное число. После того как мы реализовали
     пересчет рейтинга при добавлении отзыва, старые объявления, у которых не было отзывов, так и остались со своим "захардкоженным"
     значением (например, 4.7), что не отражает реальность.
   * Рекомендация: Создать небольшой скрипт (можно даже временный, который запускается один раз), который пройдет по всем записям в
     MarketplaceListing. Для каждого объявления он должен будет:
       1. Запросить все связанные с ним отзывы (ListingReview).
       2. Если отзывы есть, вычислить среднее значение их поля rating.
       3. Обновить поле rating в MarketplaceListing этим новым, честно вычисленным значением.
       4. Если отзывов нет, установить рейтинг в 0.
      Это синхронизирует все данные и сделает систему рейтинга консистентной.


  4. Поиск и фильтрация по городу


   * Проблема: В API для получения списка объявлений (GET /api/catalog/listings) нет возможности отфильтровать товары по городу.
     Фронтенд всегда получает все товары, хотя в интерфейсе есть фильтр по городу.
   * Рекомендация: Доработать API GET /api/catalog/listings. Он должен принимать необязательный параметр cityId. Если этот параметр
     передан, в запрос к базе данных (prisma.marketplaceListing.findMany) нужно добавить условие в блок where:


   1     where: {
   2       // ... другие условия
   3       city_id: cityId ? Number(cityId) : undefined,
   4     }
      Затем на фронтенде в FilterPanel.tsx при выборе города в фильтре нужно передавать его id в API-запрос.

  5. Безопасность паролей


   * Проблема: Пароли в базе данных хранятся в открытом виде (password: "demo123"). Это критическая уязвимость. Если кто-то получит
     доступ к базе, все пароли пользователей будут скомпрометированы.
   * Рекомендация: Перед сохранением пароля в базу его нужно хешировать. Для этого идеально подходит библиотека bcrypt.
       1. Добавить bcrypt в зависимости проекта (npm install bcrypt @types/bcrypt).
       2. При создании пользователя (auth.routes.ts) или смене пароля (profile.routes.ts) нужно хешировать пароль: const hashedPassword
          = await bcrypt.hash(password, 10);. В базу сохранять именно hashedPassword.
       3. При входе пользователя в систему нужно сравнивать введенный пароль с хешем из базы: const isValid = await
          bcrypt.compare(plainPassword, user.password);. Только если isValid равно true, пускать пользователя.
