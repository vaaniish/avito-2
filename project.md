# Карта проекта

## Назначение

Проект разделён на фронтенд, бэкенд и служебные сценарии. Основная цель текущей структуры — предсказуемое размещение файлов, чтобы по пути сразу было понятно, это страница, виджет, общий UI, доменная сущность или инфраструктурный код.

## Структура верхнего уровня

```text
.
├── backend/     # API, Prisma, доменные модули
├── frontend/    # клиентское приложение
├── scripts/     # тесты, QA, perf и служебные сценарии
├── docs/        # живая документация проекта
└── project.md   # эта краткая карта
```

## Структура фронтенда

```text
frontend/src/
├── app/         # app shell, route sync, render orchestration, глобальные app-hooks
├── pages/       # route/view entrypoints и page-private модули
├── widgets/     # переиспользуемые экранные блоки
├── entities/    # компактные доменные компоненты
├── shared/      # общие lib, ui и types
├── index.css
└── main.tsx
```

Ключевые правила:

- Всё, что соответствует экрану или route/view, лежит в `pages/`.
- Всё, что используется только одной страницей, остаётся внутри её feature-папки.
- Повторно используемые экранные блоки идут в `widgets/`.
- Небольшие доменные визуальные элементы идут в `entities/`.
- Общие примитивы, утилиты и типы лежат в `shared/`.

## Структура бэкенда

```text
backend/src/
├── lib/         # prisma, session, observability и инфраструктурные helper'ы
├── modules/     # доменные backend-модули
├── utils/       # небольшие общие backend-утилиты
├── app.ts
└── server.ts
```

Основная ось организации бэкенда — `modules/*`.

## Структура документации

```text
docs/
├── architecture/  # правила и архитектурные соглашения
└── audit/         # только актуальные аудиты и отчёты
```

## Что важно держать в голове

- Исторические audit-логи и промежуточные фазы удалены; в `docs/audit` теперь должна лежать только актуальная информация.
- Временные артефакты вроде `playwright-report/`, `test-results/` и `.DS_Store` считаются мусором и не должны попадать в репозиторий.
