# Type & Contract Audit

Date: 2026-04-20  
Scope: Stage 6, slices 1-10 (`profile` + `checkout` + `partner-listings` + `admin-complaints` + `order/payment` + `admin-transactions` + `auth/session`)

## Findings Table

| Type / Attribute | Problem | Fix | Rationale (practice/source) | Status |
|---|---|---|---|---|
| `PartnershipRequest.inn/geography/social_profile/credibility` (`String?` in Prisma) | Backend stored empty strings for optional fields instead of `null`, creating mixed semantics (`""` vs `null`) for the same missing value. | Added `toNullableTrimmedString(...)` in `profile.engagement.routes.ts` and normalized optional text inputs to `null` when empty. | Nullable fields should have one canonical "missing" representation to keep DB/API contract predictable (TypeScript strict null discipline; Prisma nullable field semantics). | fixed |
| `UserAddress.is_default` invariant | First created address could be non-default if caller omitted `isDefault`, violating expected invariant "user has a default address". | In `profile.address.routes.ts` creation flow, computed `effectiveIsDefault = isDefault || existingAddressCount === 0`, and used it for reset/create. | Domain invariant enforcement near write path; keeps state transitions deterministic and reduces edge-case bugs in UI/checkout flows. | fixed |
| API response strings in profile routes | Several static response/fallback strings in profile routes were mojibake (broken encoding), causing unstable user-facing text and forcing decode heuristics. | Replaced mojibake literals with proper UTF-8 strings in `profile.user.routes.ts`, `profile.orders.routes.ts`, `profile.engagement.routes.ts`. | Stable text contracts improve reliability and observability; avoids transport-layer ambiguity and hidden data cleanup hacks. | fixed |
| Frontend `Address` model vs backend DTO (`ProfileAddressDto`) | Frontend type was under-specified (`apartment/entrance/building/lat/lon` optional), while backend always returns deterministic values (`string` or `null`). This weakened compile-time guarantees. | Tightened `Address` type in `frontend/src/components/pages/profile.models.ts` to required contract fields and added `label` field from backend DTO. | Stronger static contracts reduce accidental undefined handling and make UI/API integration safer (TS strict type guidance). | fixed |
| `PATCH /profile/me` response typing in frontend | Frontend typed response as `{ success: boolean }`, while backend returns `{ success, user }`; this can hide server-side contract drift. | Added `ProfileUpdateResponse` type and switched `saveProfile` request typing to exact response contract in `ProfilePage.tsx`. | Explicit response typing is recommended for reliable API clients and contract-change detection (TypeScript strictness + API contract hygiene). | fixed |
| Payment/order status contract strictness (checkout) | Frontend `PaymentStatusResponse.orderStatus/paymentStatus/paymentProvider` were typed as broad `string`, hiding contract drift from backend enums. | Tightened `checkout.models.ts` with enum unions for order/transaction/provider statuses and updated `PaymentStatusResponse` accordingly. | Explicit finite unions on API clients improve change detection and reduce runtime mismatch risk (TypeScript strict typing and stable API contracts). | fixed |
| Partner listings address DTO drift | `PartnerListingsPage` had a local `ProfileAddressDto` with optional fields, drifting from canonical profile address contract. | Replaced local DTO with `Pick<Address, \"id\" \\| \"fullAddress\" \\| \"city\" \\| \"isDefault\">` from shared `profile.models.ts`. | Shared source-of-truth model prevents silent divergence across features and improves maintainability. | fixed |
| Admin complaints frontend contract strictness | `ComplaintsPage.tsx` used broad `string` fields for listing/moderation/sanction status and partial list response typing (`items/pagination` only), which could mask backend payload drift. | Tightened complaint models with explicit unions (`listingStatus`, `listingModerationStatus`, `sellerStatus`, `sanctionLevel/status`), made status-update `cascade` explicit, and fully typed list response blocks (`sort`, `filters`, `options`). | Stable API clients should model finite protocol-like fields as unions and type full response envelopes to catch drift at compile time (TypeScript strictness + API contract hygiene). | fixed |
| Checkout + partner orders transport contract drift | `CreateOrdersResponse.payment` and `DeliveryPointsResponse` in frontend were looser than backend guarantees (`payment` optional, `location/activeProvider/pagination` optional), and `PartnerOrdersPage` duplicated order-status enum + kept `tracking_provider` as broad `string \\| null`. | Tightened checkout response shapes to backend-stable fields and unified partner order status to shared `OrderStatusValue`; narrowed `tracking_provider` to known providers union. | Strongly typed response envelopes and shared enums reduce drift across payment/order flows and improve compile-time regression detection in critical checkout paths. | fixed |
| Admin transactions enum drift | `TransactionsPage.tsx` had `orderStatus`/`paymentProvider` as broad `string`, while backend returns lower-case values derived from finite enums. This weakened compile-time checks for finance-critical UI. | Replaced broad fields with `Lowercase<...>` unions built from shared checkout transport enums (`OrderStatusValue`, `TransactionStatusValue`, `PaymentProviderValue`). | Reusing shared finite enums across transaction-facing screens reduces semantic drift and improves refactor safety in operational dashboards. | fixed |
| Auth/session model (`x-user-id`) | Session auth was based on spoofable transport identity (`x-user-id` / query), which is below production security baseline. | Added bearer JWT (`HS256`) `sessionToken` issuance on `login/signup`, verification in `session` layer (`Authorization: Bearer` priority, `alg/typ/iss/aud/exp` checks), and frontend token persistence/use; legacy `x-user-id` fallback kept behind `ALLOW_LEGACY_USER_ID` compatibility gate. | Signed bearer JWTs with explicit claim verification are baseline practice for web API auth; transitional compatibility flags reduce migration risk while improving security posture incrementally. | fixed (phase 1) |
| Legacy transport identity defaults (`x-user-id` / `user_id`) | Legacy identity path still had weak defaults (`x-user-id` fallback and query identity path), which could keep insecure auth behavior available longer than intended. | Removed `query.user_id` fallback, forced production legacy mode off, and made non-production legacy mode opt-in only (`ALLOW_LEGACY_USER_ID=true`). Updated smoke suite to bearer-token-only auth checks. | Secure-by-default rollout and migration tests aligned with target auth architecture reduce spoofing risk and prevent regressions toward legacy identity paths (OWASP ASVS session/auth controls). | fixed (phase 2) |
| Production session secret policy | Session token secret had a built-in development fallback and no production strength guard, creating risk of weak or default secret usage after deployment misconfiguration. | Added fail-fast production checks in `session-token.ts` (`SESSION_TOKEN_SECRET` required in production, minimum length 32, built-in dev fallback forbidden) and startup-time validation call in `server.ts`. | Production auth secrets should be explicit, strong, and validated at startup to prevent insecure defaults (OWASP ASVS credential/session configuration and secure-by-default operations). | fixed (phase 2 hardening) |
| CI preflight gap for session config | Secure auth config checks existed in runtime, but there was no dedicated CI preflight command to block misconfigured production deployment before runtime startup. | Added `scripts/security-preflight.ts` and npm commands (`security:preflight:session`, `ci:preflight:prod-auth`) to validate production `SESSION_TOKEN_*` configuration in CI/CD. | Shift-left security validation in CI reduces deployment risk and shortens feedback loops for config errors (DevSecOps and fail-fast operations practice). | fixed |
| Legacy transport fallback still present in runtime client/server | After phased migration, backend still had optional `x-user-id` fallback and frontend transport still had emergency header fallback when token absent; this preserved spoofable identity code paths and prevented strict token-only guarantee. | Removed `x-user-id` fallback from backend `session.ts`, removed frontend request fallback header in `api.ts`, and aligned app session restore to require both stored user and token before setting authenticated state. | Strong authentication boundaries should rely on signed bearer credentials only; removing fallback identity paths reduces attack surface and operational ambiguity (OWASP ASVS session/authentication controls; secure-by-default architecture). | fixed (phase 3 token-only) |
