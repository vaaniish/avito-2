# Правила структуры фронтенда

## Цель

Фронтенд организован по feature-first схеме, чтобы путь к файлу сразу подсказывал его роль в системе.

## Основные папки

- `app/` — корневая композиция приложения, route parsing, app-shell и глобальная orchestration-логика
- `pages/` — route/view entrypoints и файлы, принадлежащие только конкретной странице
- `widgets/` — переиспользуемые экранные блоки среднего уровня
- `entities/` — небольшие доменные визуальные компоненты
- `shared/lib/` — API-клиент, поиск, метрика и общие runtime-утилиты
- `shared/ui/` — общие UI-примитивы без page ownership
- `shared/types/` — общие типы между feature-модулями

## Правила размещения

- Если файл соответствует экрану или top-level view, он лежит в `pages/`.
- Если файл нужен только одной странице, он лежит внутри её feature-папки.
- Если блок переиспользуется несколькими страницами, но сам не является страницей, он идёт в `widgets/`.
- Если компонент в основном отображает одну доменную сущность, он идёт в `entities/`.
- Если код общий и не принадлежит одной странице, он идёт в `shared/`.

## Правила нейминга

- Точки входа страниц: `PascalCase` + `Page`
- Поддерживающие feature-файлы: описательные kebab-style имена, например `checkout.hooks.ts`
- Общие UI-примитивы живут только в `shared/ui`
- Вне feature-контекста не использовать безликие имена вроде `utils.ts`, `types.ts` или `components.tsx`

## Практический фильтр

Перед созданием нового файла стоит ответить на три вопроса:

1. Это экран или route/view?
2. Это используется только одной страницей?
3. Это общая инфраструктура, UI-примитив или доменная сущность?

Ответ на эти вопросы обычно сразу подсказывает правильную папку.
